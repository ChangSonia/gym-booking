# 健身房團課報名系統

## 這是什麼

取代 LINE 群組「+1 +2」報名的系統。會員在 LINE 裡開 LIFF 網頁報名，教練在同一個 app 管理課程。

**核心觀念：報名不是聊天，是交易（transaction）。** 需要原子性、狀態機、稽核紀錄。

## 使用者

**大量長輩**（阿姨、婆婆、媽媽）。這決定了所有 UI 決策：

- 文案要短。「已報名」不是「你已經報名這堂課了」
- 按鈕大：主要按鈕 46px 高、16px 字
- 破壞性操作一律二次確認，預設選項是「不要」
- 不能做的事，不要讓他按得到（停用按鈕，而不是事後跳錯誤）

## 技術棧

- **前端 + 後端**：Next.js（App Router）
- **資料庫**：Supabase (PostgreSQL)
- **部署**：Vercel
- **身分**：LINE Login channel + LIFF

# 產品決策（已確認，不要擅自改）

這些是跟業主討論後定案的規則。要改必須先問人類。

## 報名

- **名額只增不減**。已開放報名的課，名額只能往上加。UI 上直接停用「減少」按鈕
- **候補不拆單**。剩 2 位、要報 3 人 → 整組進候補，不會「2 人上、1 人候補」
- **候補嚴格 FIFO，不跳號**。釋出 1 個名額、候補第一組是 2 人 → 等，不跳過後面的 1 人組
  - （這條是可討論的營運政策，改之前要問）
- **每筆報名最多 4 人**（`MAXQ`）
- **開放時間**：上課前 7 天中午 12:00。加開課可選「立即開放」

## 課程模板 vs 單次課程

```
Course（每週固定課模板）
   ↓ 排程器每週自動排出未來三週
Session（單次課程）
   ↑ 可個別覆寫：換教練、加名額、停課
```

**改模板只影響「指定生效日之後、且還沒開放報名的」課。**
已開放報名的課凍結，不受模板影響。改模板時要讓使用者選生效日（日期選擇器）。

理由：「這週教練請假、別人代課」不能把之後每週都改掉。

## 停課與恢復

- 停課 → 所有報名標記 `session_cancelled`，**並存下原始狀態和順位**（`prev`, `prev_position`）
- 恢復 → 只救 `session_cancelled` 的，**原樣回來、順位不變**
- 會員**自己取消**的（`cancelled`）不會被救回來
- **這兩個狀態必須分開**，共用一個 `cancelled` 是 bug 的根源

## 教練

- **教練名字所見即所得**。不做簡稱、不用正則猜
  - 曾經有個 `shortCoach()` 把「宋教練」砍成「宋」，但「雪兒老師」不砍 —— 同一份名單兩套規則，使用者輸入的東西被系統偷偷改掉。已移除
- 刪除教練 → 名下的課變「待定」，會員端也顯示「待定」（誠實，會員看到會問，反而提醒去指派）
- 教練改名 → 名下所有模板和場次一起連動

## 資料保留

- 過期的課**歸檔不刪**。標記起來不顯示，但資料留著
- 理由：刪掉報名紀錄 = 刪掉「誰常缺席」「哪堂課熱門」的證據
- 會員端只顯示本週 + 下週

## 權限

- 教練身分**必須在後端驗證**。前端說「我是教練」不算數
- 教練權限要能**設定也能移除**（離職要能撤銷）
- 第一個管理員：手動在 Supabase 後台把自己的 `is_admin` 設成 true

# 技術陷阱（踩過的坑）

## 1. 日期絕對不要用 Date 做日曆運算

**純日曆日期（哪一天、星期幾）一律用字串處理。**

```js
// ❌ 錯：getDay() 回傳的是「瀏覽器時區」的星期
new Date('2026-07-13T00:00+08:00').getDay()
// 在 UTC 瀏覽器會變成前一天，整週課表挪掉一格

// ✅ 對：UTC 錨點解析純日期
function ymd(iso){ return new Date(iso.slice(0,10)+'T00:00:00Z'); }
ymd('2026-07-13').getUTCDay()
```

只有「時間點」（開放報名的那一瞬間）才用 `Date` 加時區。

真的上線後，**會員出國、手機時區設錯，課表就會整個挪一天**。

## 2. iOS Safari 會把 19:00 變成藍色電話連結

```html
<meta name="format-detection" content="telephone=no,date=no,address=no,email=no">
```
```css
a[href^='tel'],a[x-apple-data-detectors]{color:inherit!important;text-decoration:none!important;
  pointer-events:none;font:inherit!important}
```

**這個 bug 只在真手機上出現**，電腦瀏覽器抓不到。

## 3. 併發：12:00 那一秒會超賣

```sql
BEGIN;
SELECT * FROM sessions WHERE id = ? FOR UPDATE;   -- 行鎖，關鍵
SELECT COALESCE(SUM(qty),0) FROM bookings WHERE session_id = ? AND status = 'confirmed';
-- 判斷後 INSERT
COMMIT;
```

外加 `UNIQUE(session_id, user_id)` 防重複報名（但要處理 cancelled 的情況，見 schema）。

## 4. PostgreSQL 保留字與 plpgsql 變數衝突

兩個踩過的坑（**都是跑起來才發現，parser 抓不到**）：

```sql
-- ❌ position 是保留字，在 RETURNS TABLE 裡會語法錯誤
returns table (status text, position int)

-- ❌ RETURNS TABLE 的欄位名會變成 plpgsql 變數，跟資料表欄位撞名
--    → ERROR: column reference "status" is ambiguous
returns table (status text, ...) ... select ... where status = 'confirmed'

-- ✅ 欄位改名 wl_position，回傳值加 out_ 前綴，查詢一律加 table alias
returns table (out_status text, out_position int)
... select coalesce(sum(b.qty),0) from bookings b where b.status = 'confirmed'
```

## 5. 「有效報名」要有單一定義

```js
const LIVE = ['confirmed','waitlisted'];
```

加新狀態時，散在四處的 `status !== 'cancelled'` 會全部變成錯的。**這種 bug 是加新狀態時才炸的。**

## 6. LINE 快取很頑固

改完部署後，手機要**清 LINE 快取**才看得到新版。光關掉 app 沒用。

# 設計系統

## 顏色語意（嚴格遵守）

| 顏色 | 意思 | 用在哪 |
|---|---|---|
| 藍實心 | 主要動作 | 報名、確認 |
| 深黃實心 `#8A5D00` | 主要動作，但結果是等待 | 候補 |
| 淡黃 `#FFF6E5` | 等待中的狀態 | 候補橫幅、快滿 chip |
| 淡綠 `#E4F4EA` | 成了 | 已報名卡片 |
| 紅實心 `#C8102E` | **按下去會失去東西** | 取消、停課、刪除 |
| 灰 | 次要／不可按 | 編輯、未開放 |

**紅色很稀有，只給破壞性操作。** 「重要的東西」太多了，使用者會麻痺；「按下去會失去什麼」很少見，值得一個專屬顏色。

## 按鈕層級

| 層級 | 樣式 |
|---|---|
| 卡片主動作（報名／候補／未開放） | 46px 高、16px 字 |
| 橫幅次動作（編輯） | 小尺寸、白底灰框、灰字 |
| 視窗動作（確認／取消） | 全寬、15px padding |

**同一層級內完全統一，跨層級才有差異。**

## 文案

短。以下是定案的用字：

| 情境 | 文案 |
|---|---|
| 已報名 | `✅ 已報名 2 位` / 小字 `7/16（四）20:00` |
| 候補中 | `🕒 候補中 · 1 位` / 小字 `前面尚有 0 位候補` |
| 有空位 | `剩 3 位` / 小字 `上限 12 人` |
| 額滿 | `已額滿` / 小字 `2 人候補中` |
| 未開放 | `尚未開放報名` + 倒數；最後 60 秒 `即將開放報名` |
| 按鈕 | `報名` / `候補` / `編輯` / `未開放` |
| 視窗標題 | `報名` / `候補` / `編輯報名` / `編輯候補` |
| 人數標籤 | `報名人數` / `欲候補人數` |
| 取消入口 | `取消報名` / `取消候補` |

**tab 名稱**：課表 / 我的課表 / 教練

## 週曆

- **列＝一天**，不是欄＝星期（390px 塞不下 7 欄，字會太小）
- 週一開始
- chip 底色 = 空位狀況：綠(有空) / 黃(剩3以內) / 紅(額滿) / 灰(未開放)
- chip 上不放教練名（空間不夠，點進去就看得到）
