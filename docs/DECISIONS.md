# 產品決策紀錄

每條決策都寫**為什麼**。以後想改的時候，先看看當初的理由還成不成立。

---

## 為什麼不繼續用 LINE 群組

| 問題 | 群組的結構性缺陷 |
|---|---|
| 名額計算 | 人工數 +1 +2，取消訊息穿插其中，容易錯算 |
| 公平性 | 12:00 開放，誰先發訊息取決於網速與運氣，無法驗證 |
| 取消追蹤 | 取消訊息淹沒在對話裡，候補遞補靠人眼發現 |
| 稽核 | 訊息可刪可撤回，爭議時無法還原 |
| 教練負擔 | 開課、點名、統計全靠手動 |

**核心觀念：報名不是聊天，是交易。** 需要原子性、狀態機、稽核紀錄。

---

## 名額只增不減

**決策**：已開放報名的課，名額只能往上加。

**理由**：業主說「頂多加不會減少」。這條規則讓最麻煩的情況消失了——名額 20 改 15、但已經 18 人報名，要踢誰？

**實作**：UI 上直接**停用**「減少」按鈕，不是事後跳錯誤訊息。
**不能做的事，不要讓他按得到。**

---

## 候補不拆單

**決策**：剩 2 位、要報 3 人 → 整組進候補。不會「2 人上、1 人候補」。

**理由**：長輩比較不會誤會。「我報了 3 個，怎麼只有 2 個上？」這種客訴不值得。

**代價**：門檻高一點。大團體比較難搶到。

---

## 候補嚴格 FIFO，不跳號

**決策**：釋出 1 個名額、候補第一組是 2 人 → **等**，不跳過後面的 1 人組。

**理由**：公平。先排隊的先上。

**代價**：名額會空著一陣子。

**這是營運政策，不是技術問題。** 另一種做法是「跳過塞不下的，找下一組」——效率高但團體會覺得被插隊。**改之前要問老闆。**

---

## 模板 vs 單次課程分離

**決策**：
```
Course（每週固定課模板）
   ↓ 排程器自動排出未來三週
Session（單次課程）
   ↑ 可個別覆寫
```

**理由**：「這週教練請假、別人代課」不能把之後每週都改掉。

**改模板的影響範圍**：只影響「**指定生效日之後、且還沒開放報名**」的課。

**為什麼要讓使用者選生效日**：教練心裡本來就有「從下個月開始調整」這種概念。系統幫他決定不如讓他自己選。

**已開放報名的課凍結**：已經有人報名的課，時間和名額不能被模板偷改。

---

## 停課要能恢復，而且原樣回來

**決策**：停課 → 存快照；恢復 → 原樣放回，順位不變。

**理由**：停課是健身房的錯，不該讓會員重搶。

**關鍵陷阱**：`cancelled` 這個狀態原本被兩件事共用：
- 會員自己取消
- 停課被系統退掉

**恢復時只該救後者。** 所以拆成兩個狀態：
```
cancelled          ← 會員自己取消，不會被救回來
session_cancelled  ← 停課退掉的，恢復時原樣回來
```

外加 `prev_status` / `prev_position` 存快照。

---

## 教練名字所見即所得

**決策**：不做簡稱、不用正則猜。

**歷史**：曾經有個 `shortCoach()` 把結尾的「教練」砍掉：

| 輸入 | 顯示 | 合理嗎 |
|---|---|---|
| 宋教練 | 宋 | ❌ 打了「宋教練」就是要顯示宋教練 |
| Dumars教練 | Dumars | ✅ 剛好對 |
| 雪兒老師 | 雪兒老師 | ⚠️「老師」沒被砍，所以沒縮 |

同一份名單裡兩套規則，而且**使用者親手改了名字，系統還是不理他**。

**教訓：使用者輸入的東西，系統不要偷偷改。** 想要簡稱就開一個簡稱欄位讓人填，不要用正則猜。

這個 bug 用假資料看起來很聰明，換上真資料立刻變成錯的。

---

## 刪除教練 → 課變「待定」

**決策**：可以刪，名下的課變「待定」，**會員端也顯示「待定」**。

**理由**：誠實。會員看到「待定」會問，反而提醒教練去指派。

**替代方案**（沒採用）：擋住不讓刪，要求先改教練。太麻煩。

---

## 過期資料歸檔不刪

**決策**：`archived = true`，不 DELETE。

**理由**：刪掉報名紀錄 = 刪掉「誰常缺席」「哪堂課最熱門」的證據。

儲存成本幾乎是零（一年的報名紀錄大概幾 MB），但**資料刪了就回不來**。

---

## 教練權限在後端驗證

**決策**：前端說「我是教練」不算數。

**理由**：任何人都能改瀏覽器裡的東西。

**流程**：
```
1. 前端送 LINE ID Token
2. 後端向 LINE 驗證 → 拿到可信的 line_user_id
3. 查 users 表 → 這人 is_coach 嗎？
```

**絕對不要**直接相信前端傳來的 `userId`。

**第一個管理員的雞生蛋問題**：你自己登入一次，然後手動在 Supabase 把 `is_admin` 設成 true。

# UI 決策

## 使用者是長輩

這決定了一切：

- **文案要短**：「已報名」不是「你已經報名這堂課了」
- **按鈕要大**：主要按鈕 46px 高、16px 字（一根手指按得到）
- **二次確認**：破壞性操作要按兩次，預設選項是「不要」
- **不能做的事不要讓他按得到**

## 顏色語意

| 顏色 | 意思 |
|---|---|
| 藍實心 | 主要動作 |
| 深黃實心 | 主要動作，但結果是等待（候補） |
| 淡黃 | 等待中的狀態 |
| 淡綠 | 成了 |
| 紅實心 | **按下去會失去東西** |
| 灰 | 次要／不可按 |

**紅色很稀有，只給破壞性操作**（取消、停課、刪除）。

**理由**：「重要的東西」太多了，使用者會麻痺。但「按下去會失去什麼」很少見，值得一個專屬顏色。

## 按鈕層級

**同一層級內完全統一，跨層級才有差異。**

「報名」永遠是那頁最好按的東西，「編輯」不能跟它搶。

曾經試過「全系統一種尺寸」，結果「取消」跟「報名」一樣顯眼，誤觸機率變高。

## 週曆用「列＝一天」不是「欄＝星期」

**理由**：390px 寬的手機，7 欄的話一欄只剩 70px，字會小到長輩看不了。

海報可以用欄＝星期（尺寸大、不用點），手機不行。

## 週曆 chip 不放教練名

**理由**：空間不夠。點進去就看得到。

課名優先，教練名次之。

# 沒做但記著的事

## 場地衝堂

現在允許任何兩堂課排同一時間。週四 19:00 和 19:10 是刻意的（不同場地），但系統不會檢查。

哪天有人手滑排了三堂同時段，系統照收。要擋要加 `room` 欄位 + 衝堂檢查。

## no-show 統計

資料模型留了空間（`bookings` 有完整的歷史），功能還沒做。

這是團課系統的關鍵——否則有人佔位不來。

## 「滿 3 人才開班」

附近的瑜伽教室有這種模式（COMING SOON = 揪團中）。

這是**完全不同的狀態機**：先收報名 → 到門檻才成立 → 不到就退掉。不是加個欄位的事。

**業主目前的課表沒有這種課**，所以沒做。

## LINE MINI App

官方說未來 LIFF 會併入 LINE MINI App，建議新專案直接做 MINI App。

但 MINI App **要審核**，流程長很多。先用 LINE Login，之後要遷移的話程式碼幾乎不用改，只是換 channel。
